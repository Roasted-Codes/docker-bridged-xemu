HaloCaster Offset Analysis — Is the "Linux Approach" Bullshit?
================================================================
Date: 2026-02-12
Context: Evaluating claims from a Discord conversation where Claude
         suggested HaloCaster offsets are portable across OS/builds.

================================================================
WHAT THE FRIEND'S CLAUDE GETS RIGHT
================================================================

The HaloCaster mechanism is accurately described. HaloCaster does this:

  1. gva2gpa(0x80000000) -> guest physical address of Xbox RAM base
  2. gpa2hva(that)        -> HOST VIRTUAL ADDRESS in xemu.exe's process memory
  3. Adds 0x5C000 + offset as a flat offset
  4. ReadProcessMemory() on xemu.exe at that host VA

HaloCaster NEVER uses QMP "xp". It uses QMP purely for address translation,
then reads via Windows process memory APIs. StatsBorg's own REFERENCE_SUMMARY.md
in the HaloCaster repo confirms: "QMP-only -- none of this is reachable via XBDM."

================================================================
WHAT THE FRIEND'S CLAUDE GETS WRONG
================================================================

"Copy the offsets directly -- they work on any OS" -- this is the bullshit part.

The claim that offsets are universal "Xbox game addresses" ignores what we already
documented in CLAUDE.md: these addresses read as ALL ZEROS on our CerbiosDebug /
docker-bridged-xemu setup. HaloCaster's own resolve_addresses() error message says:

    "the game region/version may be incompatible"

The offsets aren't BIOS-independent. They depend on how the XBE is loaded, how
the kernel maps memory, and which build you're running. On CerbiosDebug, physical
addresses 0x036090EC (variant_info), 0x0360BF02 (game_stats), etc. contain zeros.
The data structures simply aren't populated there.

================================================================
THE REAL DIFFERENCE NOBODY IS SEEING
================================================================

StatsBorg and HaloCaster read memory through fundamentally different paths:

                    HaloCaster                  StatsBorg
  Translation:      gva2gpa + gpa2hva           gva2gpa (user VA) or
                                                 strip high bit (kernel VA)
  Memory read:      ReadProcessMemory on         QMP "xp" (guest physical
                    xemu.exe host process         memory read)
  Access model:     Host process memory          Guest physical memory
  Requires:         Running on same machine      Network access to QMP only
                    as xemu

StatsBorg uses "xp" which reads guest physical RAM as QEMU exposes it.
HaloCaster uses ReadProcessMemory which reads xemu's host process virtual
memory. For standard RAM these SHOULD be identical -- but the point is moot
because the data isn't at those offsets on our build.

================================================================
THE LINUX process_vm_readv() PROPOSAL
================================================================

The friend's process_vm_readv() approach is technically sound -- it IS the
Linux equivalent of HaloCaster's ReadProcessMemory(). But it requires:

  1. Running on the SAME MACHINE as xemu (not Docker-bridged over network)
  2. The offsets ACTUALLY CONTAINING VALID DATA (they don't on CerbiosDebug)

So it's a correct Linux port of HaloCaster's read mechanism, but it won't
solve our problem because the problem isn't the read mechanism -- it's that
the data isn't there.

================================================================
WHAT'S ACTUALLY USEFUL FROM THE CONVERSATION
================================================================

One thing buried in the noise: PATTERN SCANNING PHYSICAL RAM. If HaloCaster's
structures exist at DIFFERENT offsets on CerbiosDebug, a targeted scan during
known game states (e.g., mid-game with 1 kill, or during a CTF game) could
locate them. research/scan_gametype.py already does something similar.

The other useful insight: HaloCaster's LINEAR MAPPING TRICK --
Translate(0x80000000) + 0x5C000 -- bypasses Xbox page tables entirely. This
is why HaloCaster can read addresses in the kernel VA gap that XBDM returns
404 for. StatsBorg's QMP "xp" command also bypasses page tables (it reads
physical), so we already have equivalent access. The data just isn't there.

================================================================
HALOCASTER resolve_addresses() FLOW (from source)
================================================================

  Step 1: Define offsets (relative to host base):
    game_stats      = 0x35ADF02
    variant_info    = 0x35AD0EC
    medal_stats     = 0x35ADF4E
    life_cycle      = 0x35E4F04
    session_players = 0x35AD344
    tags            = 0x360558C
    ... etc

  Step 2: Try LINEAR MAPPING:
    host_base = Translate(0x80000000) + 0x5C000
    For each offset: address = host_base + offset
    Test: ReadProcessMemory(tags_address) -- if non-zero, success

  Step 3: If linear fails, try PER-ADDRESS TRANSLATION:
    For each offset:
      xbox_va = 0x8005C000 + offset
      address = Translate(xbox_va)   // gva2gpa + gpa2hva per-address
    Re-test tags value

  Step 4: Resolve game state pointers:
    Read pointer values from resolved addresses
    Translate those pointers to get second-level host addresses

  Step 5: All subsequent reads via ReadProcessMemory at resolved host VAs

================================================================
TL;DR
================================================================

The friend's Claude understands HaloCaster's mechanism correctly but draws
the wrong conclusion. The offsets aren't portable. We already knew that --
we've spent hours proving it. The read mechanism (xp vs ReadProcessMemory /
process_vm_readv) doesn't matter because the problem is the addresses, not
how you read them.
